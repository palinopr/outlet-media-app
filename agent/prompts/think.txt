### Priority 4: Business Monitoring
- **Step 0: Freshness check** — session cache files are bare JSON arrays with NO `scraped_at` wrapper. Check the file modification time: `ls -la session/last-campaigns.json`. If mtime is older than 24 hours, log "stale data — last sync was [mtime]" in LEARNINGS.md. Monitoring results may be unreliable.
- Check session/last-campaigns.json (last Meta pull) — any anomalies?
  - ROAS < 2.0 on any ACTIVE campaign is a flag (ignore PAUSED campaigns with low ROAS — they're already off)
  - Spend spiking or dropping unexpectedly is a flag
  - **Daily pacing check (do this every P4 cycle):**
    - Session cache has `spend` in DOLLARS and `daily_budget_cents` in CENTS
    - **Skip pacing** for any campaign where `daily_budget_cents` is null or 0 — can't calculate pacing without a budget
    - If `start_time` is missing from cache AND Supabase, **skip pacing** for that campaign and log it. Do NOT block the whole monitoring cycle.
    - If `start_time` is missing from cache only, query Supabase: GET meta_campaigns?select=campaign_id,start_time&status=eq.ACTIVE
    - Calculate days_elapsed = (today - start_time) in days (minimum 1)
    - expected_spend_dollars = (daily_budget_cents / 100) * days_elapsed
    - pacing_ratio = spend / expected_spend_dollars
    - Flag if pacing_ratio > 1.3 → "Campaign X is overpacing (+30%)"
    - Flag if pacing_ratio < 0.7 → "Campaign X is underpacing (-30%)"
    - Only flag ACTIVE campaigns (status === "ACTIVE")
    - **Pacing limitation**: start_time-based pacing is unreliable for campaigns with pause history (e.g., KYBBA was paused Dec-Feb). The formula assumes continuous spending since start_time, so paused periods inflate expected spend. For campaigns with known pause history: SKIP raw pacing and note "skipped — pause history." Preferred alternative (when available): compute spend deltas from campaign_snapshots over 3+ day spans for actual run rate (consecutive-day deltas are unreliable due to Meta reporting lag).
  - **ROAS trend check (if snapshots exist):**
    - Query Supabase: GET campaign_snapshots?campaign_id=eq.{ID}&order=snapshot_date.desc&limit=7
    - **Column naming**: snapshots use `spend` (NOT `spend_cents`) — value is in cents. `roas` is a float. See MEMORY.md for full column listing.
    - If ROAS is declining for 3+ consecutive snapshot dates on an ACTIVE campaign, flag it
    - **Important**: consecutive-day ROAS changes may be noise due to Meta reporting lag. Trend is only meaningful when snapshot dates are 3+ days apart OR when the same direction persists across 3+ snapshot dates regardless of spacing.
    - If no snapshots exist yet, skip this check silently
  - **Marginal ROAS check (requires 2+ snapshots for the same campaign):**
    - For each ACTIVE campaign with 2+ snapshots, calculate incremental return:
    - spend_dollars = spend_cents / 100; Revenue = spend_dollars × roas (for each snapshot date)
    - Marginal ROAS = (Revenue_new - Revenue_old) / (spend_dollars_new - spend_dollars_old)
    - Flag if marginal ROAS < 1.0 → "Campaign X losing money on new spend (marginal ROAS = Y×)"
    - Marginal ROAS is MORE actionable than blended ROAS — a campaign can have 3.0× blended but 0.5× marginal, meaning it's coasting on historical performance while current spend is unprofitable
    - If blended ROAS is above 2.0 but marginal is below 1.0, project when blended will cross 2.0: additional_spend = (Revenue - 2.0 × Spend) / (2.0 - marginal_roas). Days to cross = additional_spend / daily_budget
    - Caveat: **consecutive-day** snapshot deltas are unreliable due to Meta intraday reporting lag — campaigns show <3% of expected daily spend in next-day deltas (e.g., $0.22-$1.43 on $100/day budgets). **Prefer 3+ day gaps** between snapshots for meaningful marginal analysis. Multi-day calculations (e.g., Feb 19→24) are the most reliable.
- **Campaign status change detection:**
    - Compare LEARNINGS.md notes from previous cycles against current session cache
    - If a campaign was ACTIVE last cycle but is now PAUSED (or vice versa), note it
    - ACTIVE→PAUSED is informational (Jaime likely paused it manually)
    - PAUSED→ACTIVE is a flag — confirm Jaime intended to reactivate (real money starts flowing)
    - New campaigns not seen in previous cycles should be noted
- Check session/last-events.json (last TM One pull) — any anomalies?
  - If the file doesn't exist (TM One not yet configured), skip with a note in LEARNINGS.md — don't error
  - Also check for other TM-related files: session/tm-v2-output.json (scraped event data), session/tm-*-stderr.log (error logs from headless browser scraping). If these exist, note their mtime and any errors found in stderr logs.
  - Slow ticket velocity on an upcoming show is a flag
  - Status changes (on_sale → cancelled) are flags
- If you find a real anomaly, do TWO things:
  1. POST the alert to the dashboard via:
     ```

     curl -s -X POST "$BASE_URL/api/alerts" \

     ```
     level must be one of: "info", "warning", "critical"
     Note: the alerts endpoint accepts { secret, message, level } only — no client_slug field.
  2. Draft a brief Telegram message (see Communication below)
