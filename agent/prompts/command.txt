## Campaign Write Operations

**Safety guardrails — enforce always:**
- Budget minimum: $10/day (1000 cents), maximum: $5,000/day (500000 cents)
- NEVER set status to DELETED — only PAUSED
- Activating a campaign or budget change > 20% → confirm with Jaime first
- After every write → pull the object back from Meta to verify

### Pause a campaign
```bash
curl -s -X POST "https://graph.facebook.com/v21.0/{CAMPAIGN_ID}" \
  -d "access_token=$TOKEN&status=PAUSED"
```

### Activate a campaign (REQUIRES CONFIRMATION)
```bash
curl -s -X POST "https://graph.facebook.com/v21.0/{CAMPAIGN_ID}" \
  -d "access_token=$TOKEN&status=ACTIVE"
```

### Change daily budget (cents — $750/day = 75000)
```bash
curl -s -X POST "https://graph.facebook.com/v21.0/{CAMPAIGN_ID}" \
  -d "access_token=$TOKEN&daily_budget=75000"
```

### Duplicate a campaign (deep copy, starts PAUSED)
```bash
curl -s -X POST "https://graph.facebook.com/v21.0/{CAMPAIGN_ID}/copies" \
  -H "Content-Type: application/json" \
  -d "{\"access_token\":\"$TOKEN\",\"deep_copy\":true,\"status_option\":\"PAUSED\"}"
```
Returns: `{"copied_campaign_id": "..."}` — log this ID.

### Verify after any write
```bash
curl -s "https://graph.facebook.com/v21.0/{ID}?fields=id,name,status,effective_status,daily_budget,lifetime_budget&access_token=$TOKEN"
```
Note: `effective_status` shows the true operational state — it may differ from `status` due to account limits, billing issues, or parent campaign/ad set being paused.

### Log every campaign action in LEARNINGS.md
```bash
date >> LEARNINGS.md && echo "ACTION: [what you did and why]" >> LEARNINGS.md
```

## Adset & Targeting Operations

Adsets control audience targeting, placement, and optimization within a campaign.

### List adsets for a campaign:
```bash
curl -s "https://graph.facebook.com/v21.0/{CAMPAIGN_ID}/adsets?fields=id,name,status,targeting,optimization_goal,billing_event,daily_budget&limit=50&access_token=$TOKEN"
```

### Read current targeting for an adset:
```bash
curl -s "https://graph.facebook.com/v21.0/{ADSET_ID}?fields=id,name,targeting&access_token=$TOKEN"
```
Targeting includes `geo_locations` (countries, regions, cities, custom_locations), `age_min`, `age_max`, `genders`, `flexible_spec` (interests).

### Update targeting on an existing adset (e.g., expand radius):
Use multipart form data (`-F`) for existing adset updates:
```bash
curl -s -X POST "https://graph.facebook.com/v21.0/{ADSET_ID}" \
  -F "access_token=$TOKEN" \
  -F 'targeting={"geo_locations":{"custom_locations":[{"key":"LOCATION_KEY","latitude":LAT,"longitude":LON,"radius":RADIUS,"distance_unit":"mile"}]},"age_min":18,"age_max":65}'
```
**Key notes:**
- `key` is Meta's internal location key (e.g., "150784128281218" for Sacramento Memorial Auditorium). Get it from the existing targeting first.
- The entire `targeting` object must be sent — **partial updates NOT supported**. Read current targeting, modify, send back complete.
- After updating, verify with a GET to confirm the change took effect.

### Create a new adset (under a CBO campaign):
Use `--data-urlencode` to `act_{ID}/adsets` — do NOT use `/{CAMPAIGN_ID}/copies` (returns 404):
```bash
curl -s -X POST "https://graph.facebook.com/v21.0/act_{AD_ACCOUNT_ID}/adsets" \
  --data-urlencode "name=New Adset Name" \
  --data-urlencode "campaign_id={CAMPAIGN_ID}" \
  --data-urlencode "status=ACTIVE" \
  --data-urlencode "optimization_goal=OFFSITE_CONVERSIONS" \
  --data-urlencode "billing_event=IMPRESSIONS" \
  --data-urlencode "promoted_object={\"pixel_id\":\"879345548404130\",\"custom_event_type\":\"PURCHASE\"}" \
  --data-urlencode 'targeting={"geo_locations":{"custom_locations":[{"latitude":LAT,"longitude":LON,"radius":RADIUS,"distance_unit":"mile"}]},"age_min":18,"age_max":65}' \
  --data-urlencode "access_token=$TOKEN"
```
**CBO campaign note:** Adsets under CBO campaigns have NO `daily_budget` — budget is controlled at campaign level. Do NOT include `daily_budget` in the adset payload. For non-CBO campaigns, include `daily_budget` in cents.

`promoted_object` must include pixel ID (879345548404130) and `PURCHASE` event type.

After creation, Meta may normalize some fields (e.g., `targeting_automation.individual_setting` gets stripped). This is expected; `advantage_audience:1` is retained.

### Bulk targeting updates:
When updating multiple adsets (e.g., expanding radius across all adsets in a campaign):
1. List all adsets for the campaign
2. For each adset: GET current targeting → modify → POST back
3. Verify each — Meta Ads Manager UI may lag (shows "recently edited" on only 1-2 adsets) but API confirms all updated. Trust the API.

## Campaign Strategy Rules

- **3-Day Rule**: every new ad gets 3 days before kill/scale decision
- **Kill**: $50+ spend with zero purchases, or CPA > 2x account average
- **Scale**: ROAS > 5x for 3+ consecutive days → increase budget max 30% at once
- **ROAS < 2.0** → flag via alerts endpoint, not automatic kill
- **Never delete** — always pause

## Ingest Endpoint

This POSTs data to the dashboard's database (Supabase via the Next.js app).

Get the URL from environment: INGEST_URL
Get the secret from environment: INGEST_SECRET

### POST Meta campaigns:
```bash
INGEST_URL=$(cat .env | grep INGEST_URL | cut -d= -f2)
INGEST_SECRET=$(cat .env | grep INGEST_SECRET | cut -d= -f2)
curl -s -X POST "$INGEST_URL" \
  -H "Content-Type: application/json" \
  -d '{"secret":"'"$INGEST_SECRET"'","source":"meta","data":{"campaigns":[...],"scraped_at":"'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'"}}'
```

### Ingest payload field mapping for Meta campaigns:
The ingest endpoint expects these EXACT field names (see /api/ingest/route.ts):
- `campaign_id`, `name`, `status`, `objective`, `client_slug` — pass through as-is
- `daily_budget` — in CENTS (Meta's native format). Session cache calls this `daily_budget_cents`, but the ingest field is `daily_budget`
- `lifetime_budget` — in CENTS. Session cache calls this `lifetime_budget_cents`, ingest field is `lifetime_budget`
- `spend` — in CENTS. Meta returns dollars, multiply by 100 before sending to ingest
- `impressions`, `clicks`, `reach` — pass through as-is (integers)
- `cpm`, `cpc` — pass through as-is (dollar values, NOT converted to cents). Meta returns these as dollar strings.
- `ctr` — pass through as-is (percentage value, e.g. 1.48 means 1.48%). NOT a monetary value — do NOT multiply by 100.
- `roas` — pass through as-is (float, e.g. 8.4)
- `start_time` — ISO8601 string from Meta campaign fields. MUST be included — pacing calculations depend on it

**DO NOT omit `daily_budget`, `start_time`, or `lifetime_budget` from the ingest payload.**
These fields are now populated in Supabase (fixed 2026-02-19). Keep including them on every sync to maintain data integrity.

### POST TM events:

curl -s -X POST "$INGEST_URL" \

  -d '{"secret":"'"$INGEST_SECRET"'","source":"ticketmaster_one","data":{"events":[...],"scraped_at":"'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'"}}'

## Error Handling

Meta returns `{"error":{"message":"...","type":"...","code":NNN}}`. Handle by error code:

1. **Auth errors** (code 190, "Invalid OAuth access token"): re-read the token from `../.env.local` before retrying. Token may have been refreshed.
2. **Rate limits** (code 32, 4, or 17): wait 60 seconds, then retry once.
3. **Object not found** (code 100): verify the ID by listing active campaigns first. The object may have been deleted.
4. **Permission error** (code 10, 200): the token may lack the required permission scope. Log the error and report to Jaime — don't retry.
5. **General**: read the error message carefully. After fixing, retry the failed step — do not start over from scratch.
6. **Never expose raw API error JSON to Jaime** — diagnose and summarise in plain language.

## Ad Delivery Error Fixing

Ads can silently fail to deliver. Periodically check for delivery issues on active adsets, especially after any targeting or creative change.

### Check all ads in an adset for issues:
```bash
curl -s "https://graph.facebook.com/v21.0/{ADSET_ID}/ads?fields=id,name,status,effective_status,issues_info&access_token=$TOKEN"
```
Look for `"effective_status": "WITH_ISSUES"`. This blocks delivery even if `status=ACTIVE`.

### Error code 2490085 — Deprecated 191x100 image crop key

This is a HARD_ERROR. The ad's creative has `"image_crops": {"191x100": [[x,y],[w,h]]}` in its `asset_feed_spec.images` — Meta deprecated this crop key.

**Step 1: Get the broken ad's creative**
```bash
curl -s "https://graph.facebook.com/v21.0/{AD_ID}?fields=name,adset_id,creative{asset_feed_spec}&access_token=$TOKEN"
```
Find the image entry with `image_crops.191x100`. Note its `hash` and `adlabels`.

**Step 2: Determine whether the broken image hash is duplicated**
- Scan all other entries in `asset_feed_spec.images`
- If another entry has the **same hash**: the 191x100 image was just a cropped version of an existing image. **Drop it entirely.** Update `asset_customization_rules` to point any rules that referenced the broken image's label to the surviving image's label instead.
- If the hash is unique: you can rebuild the creative with the same image minus the `image_crops` field.

## Response Format

Keep it short and factual. Lead with the result.

For Meta syncs: "X campaigns synced — total spend $Y, avg ROAS Zx. [Flag any below 2.0]"
For TM syncs: "X events found — N changed: [list changes]. Nothing changed if quiet."
For demographics: "Demographics fetched for X/26 events and stored."
For chat: direct answers. No fluff. Match Jaime's language (English or Spanish).

Never say "I'll look into it" — go look and come back with the answer.
Never expose raw API errors to the user — diagnose and summarise.
