## Delegation

You can delegate tasks to other agents using structured JSON blocks in your response. The system scans for these blocks and routes them through the task queue.

**Format:** Include a fenced JSON code block with a `"delegate"` key:

```json
{"delegate": "media-buyer", "action": "check-roas", "params": {"campaign": "KYBBA Miami", "days": 7}}
```

**Available targets:**
- `media-buyer` -- Meta Ads tasks (pull ROAS, change budgets, analyze campaigns)
- `tm-agent` -- Ticketmaster tasks (event data, demographics, zip codes)
- `creative` -- Creative review tasks (ad analysis, copy feedback)
- `reporting` -- Analytics tasks (cross-source reports, trend analysis)
- `client-manager` -- Client relationship tasks
- `don-omar-agent` -- EATA/Vivaticket tasks (Don Omar BCN ticket data, velocity, revenue)
- `email-agent` -- Email tasks (check inbox, draft replies, label/organize, contact lookup)
- `meeting-agent` -- Calendar and Google Meet tasks (schedule, move, cancel, inspect meetings)

**How it works:**
1. You write your response including one or more JSON delegation blocks
2. The system extracts each block and enqueues it in the task queue
3. The target agent receives the task and executes it in their channel
4. Results are posted to the target channel and logged to #agent-feed

**Example:**
```
KYBBA is tanking. Let me get a full picture.

\`\`\`json
{"delegate": "media-buyer", "action": "adset-analysis", "params": {"campaign": "KYBBA Miami", "metric": "ROAS by adset last 7d"}}
\`\`\`

\`\`\`json
{"delegate": "reporting", "action": "cross-source-report", "params": {"client": "kybba", "period": "last 7 days"}}
\`\`\`
```

**Rules for delegation:**
- Only delegate when the task genuinely needs another agent's specialized tools
- For simple data pulls you can do yourself (Meta API, Supabase), just do it directly
- Max 5 delegations per response, max depth 2 (an agent you delegate to can delegate once more)
- The `action` field describes what to do; `params` provides context
- When work started in another human-facing channel and the specialist should report back there, include `"replyChannel": "<channel-name>"` in `params`.

### Parsing Meta data
- `purchase_roas[0].value` -> ROAS (string -> float)
- Revenue = spend x ROAS
- CPA = spend / purchases (actions where action_type = "purchase")
- Budgets from Meta are in CENTS (divide by 100)
- **Supabase column naming gotchas:**
  - `meta_campaigns`: name column is `name` (NOT `campaign_name`)
  - `campaign_snapshots`: spend column is `spend` — value is in CENTS despite the name (NOT `spend_cents`)
  - `agent_tasks`: use `from_agent`, `to_agent`, `action`, `status`, and `created_at` for work routing and ordering
  - `agent_runtime_state`: use key-value rows for heartbeat/watch state (for example `heartbeat`, `gmail_watch`)
  - `agent_alerts`: level must be "info"/"warn"/"error" (Zod-validated)
  - `tm_events`: name column is `name` (NOT `event_name`), date column is `date` (NOT `event_date`)
  - `event_snapshots`: event identifier is `tm_id` (NOT `event_id`). Columns: tm_id, tickets_sold, tickets_available, gross, snapshot_date
  - `tm_event_demographics`: fan demographics by event (fans_total, age breakdowns). Join on tm_id.

## Marginal ROAS (for scale/kill decisions)

Blended ROAS can mask declining performance. For any "should I scale?" or "should I kill?" question, compute marginal ROAS from `campaign_snapshots`:

```
Revenue = (spend_cents / 100) * roas   (for each snapshot date)
Marginal ROAS = (Revenue_new - Revenue_old) / (Spend_new - Spend_old)
```

- **Marginal > 2.0x** -> safe to scale
- **Marginal 1.0-2.0x** -> hold, monitor
- **Marginal < 1.0x** -> losing money on new spend, recommend pause or restructure
- Need 3+ day gap between snapshot dates for reliable results (consecutive-day deltas are noise due to Meta reporting lag)
