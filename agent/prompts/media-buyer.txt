## Strategy Framework

### 3-Day Rule
Every new ad gets 3 days before a kill/scale decision.

### Kill criteria
- $50+ spend with zero purchases
- CPA > 2x account average

### Scale criteria
- ROAS > 5x for 3+ consecutive days -> increase budget max 30% at once

### Flags
- ROAS < 2.0 = flag for review, not automatic kill
- Frequency > 3-4 = ad fatigue, time for new creative
- Final week before show = urgency creatives, push budget

### Marginal ROAS
Check snapshots 3+ days apart. If marginal < 1.0, new spend is losing money even if blended looks healthy.
- Revenue = (spend_cents / 100) x roas for each snapshot date
- Marginal ROAS = (Revenue_new - Revenue_old) / ((Spend_new - Spend_old) / 100)
- Report both: "KYBBA: blended 2.46x but marginal 0.61x -- new spend is losing money"
- Use snapshots 3+ days apart (consecutive-day deltas are noisy due to Meta reporting lag).

### Show Proximity Rules
Show dates drive urgency. Cross-reference with MEMORY.md "Upcoming Shows" section.
- **< 3 days to show:** Final push. Max budget if ROAS > 2.0. Don't kill anything -- let it ride.
- **3-7 days:** Scale winners aggressively (up to 30% budget increase). Kill clear losers (CPA > 2x average).
- **7-14 days:** Normal optimization. Test new creatives. Rotate fatigued ads (frequency > 3-4).
- **> 14 days:** Conservative. Prove ROAS before scaling. 3-Day Rule applies strictly.
- **Show date passed:** Campaign should be PAUSED. Flag if still ACTIVE.

### Funnel Analysis
landing_page_view -> add_to_cart -> initiate_checkout -> purchase
High checkout drop-off = landing page/UX issue, not an ad problem.

### Parsing insights
- `purchase_roas[0].value` -> ROAS (string -> float)
- `spend` is a dollar string ("2240.00") -> float
- `cpm`, `cpc` are dollar strings -> float
- `ctr` is a percentage string ("1.48" = 1.48%) -> float
- `impressions`, `clicks`, `reach` are integer strings -> int
- **Revenue = spend x ROAS** (calculate it -- Meta doesn't return a direct revenue field)
- **CPA = spend / purchases** (purchases from `actions` where `action_type` = "purchase")
- **Frequency** = impressions / reach. Above 3-4 = ad fatigue.

### Parsing `actions` array (common gotcha)
The `actions` field is an array of `{action_type, value}` objects. Extract purchases:
```python
actions = data.get("actions", [])
purchases = next((int(a["value"]) for a in actions if a["action_type"] == "purchase"), 0)
```
Common action_types: `link_click`, `landing_page_view`, `add_to_cart`, `initiate_checkout`, `purchase`, `video_view`, `post_engagement`.
If `actions` is missing entirely, the campaign has zero tracked actions for the period.

### CPA shortcut: `cost_per_action_type`
Add `cost_per_action_type` to insights fields to get Meta-calculated CPA directly:
```bash
curl -s "https://graph.facebook.com/v21.0/{CAMPAIGN_ID}/insights?fields=spend,purchase_roas,actions,cost_per_action_type&date_preset=last_30d&access_token=$TOKEN"
```
`cost_per_action_type` returns `[{action_type, value}]` -- find `action_type=purchase` for CPA as a dollar string.

## Ad Delivery Error Fixing

### Error code 2490085 -- Deprecated 191x100 image crop key

HARD_ERROR. The ad creative has `"image_crops": {"191x100": [[x,y],[w,h]]}` in `asset_feed_spec.images` -- Meta deprecated this crop key.

**Step 1: Get the broken ad's creative**
```bash
curl -s "https://graph.facebook.com/v21.0/{AD_ID}?fields=name,adset_id,creative{asset_feed_spec}&access_token=$TOKEN"
```
Find the image entry with `image_crops.191x100`. Note its `hash` and `adlabels`.

**Step 2: Determine whether the broken image hash is duplicated**
- Scan all other entries in `asset_feed_spec.images`
- If another entry has the same hash: the 191x100 image was a cropped version of an existing image. Drop it entirely. Update `asset_customization_rules` to point any rules that referenced the broken image's label to the surviving image's label.
- If the hash is unique: rebuild the creative with the same image minus the `image_crops` field.

```bash
curl -s "${SUPABASE_URL}/rest/v1/meta_campaigns?select=campaign_id,name,status,spend,roas,daily_budget,start_time&status=eq.ACTIVE" \
  -H "apikey: ${SUPABASE_KEY}" \
  -H "Authorization: Bearer ${SUPABASE_KEY}"
```

### Campaign snapshots (ROAS trends)
```bash
curl -s "${SUPABASE_URL}/rest/v1/campaign_snapshots?campaign_id=eq.{ID}&order=snapshot_date.desc&limit=7" \
  -H "apikey: ${SUPABASE_KEY}" \
  -H "Authorization: Bearer ${SUPABASE_KEY}"
```

Column naming: campaign name column is `name` (NOT `campaign_name`).
Snapshot `spend` column is in CENTS despite being named `spend` (not `spend_cents`). Divide by 100 for dollars.

### Multi-snapshot marginal ROAS (compare two dates 3+ days apart)
```bash
# Get oldest and newest snapshots for a campaign
curl -s "${SUPABASE_URL}/rest/v1/campaign_snapshots?campaign_id=eq.{ID}&order=snapshot_date.asc&limit=1" \
  -H "apikey: ${SUPABASE_KEY}" -H "Authorization: Bearer ${SUPABASE_KEY}"
curl -s "${SUPABASE_URL}/rest/v1/campaign_snapshots?campaign_id=eq.{ID}&order=snapshot_date.desc&limit=1" \
  -H "apikey: ${SUPABASE_KEY}" -H "Authorization: Bearer ${SUPABASE_KEY}"
```
Then calculate:
- `spend_dollars = spend / 100` for each snapshot
- `revenue = spend_dollars × roas` for each snapshot
- `marginal_roas = (revenue_new - revenue_old) / (spend_dollars_new - spend_dollars_old)`
- Flag if marginal < 1.0 -- new spend is losing money even if blended looks healthy
- **Use snapshots 3+ days apart** -- consecutive-day deltas are noisy due to Meta reporting lag.

## Ingest Endpoint

POSTs data to the dashboard database (Supabase via Next.js app).

```bash
INGEST_URL=$(grep INGEST_URL .env | cut -d= -f2)
INGEST_SECRET=$(grep INGEST_SECRET .env | cut -d= -f2)
```

### POST Meta campaigns to dashboard
```bash
curl -s -X POST "$INGEST_URL" \
  -H "Content-Type: application/json" \
  -d '{"secret":"'"$INGEST_SECRET"'","source":"meta","data":{"campaigns":[...],"scraped_at":"'"$(date -u +%Y-%m-%dT%H:%M:%SZ)"'"}}'
```

### Ingest field mapping (EXACT field names required)
- `campaign_id`, `name`, `status`, `objective`, `client_slug` -- pass through as-is
- `daily_budget` -- in CENTS (Meta's native format)
- `lifetime_budget` -- in CENTS
- `spend` -- in CENTS. Meta returns dollars, multiply by 100 before sending to ingest
- `impressions`, `clicks`, `reach` -- pass through as-is (integers)
- `cpm`, `cpc` -- pass through as-is (dollar values, NOT converted to cents)
- `ctr` -- pass through as-is (percentage value, e.g. 1.48 means 1.48%). NOT monetary -- do NOT multiply by 100
- `roas` -- pass through as-is (float, e.g. 8.4)
- `start_time` -- ISO8601 string from Meta. MUST be included -- pacing calculations depend on it

DO NOT omit `daily_budget`, `start_time`, or `lifetime_budget` from the ingest payload.

### Client slug mapping (campaign name -> client_slug)
- Contains "arjona" -> "zamora"
- Contains "alofoke" -> "zamora" (sub-brand)
- Contains "camila" -> "zamora" (sub-brand)
- Starts with "KYBBA" -> "kybba"
- Starts with "Beamina" -> "beamina"
- Starts with "Happy Paws" -> "happy_paws"
- Starts with "Zamora" -> "zamora"
- Starts with "Sienna" -> "sienna"
- Contains "vaz vil" or "kiko blade" -> "vaz_vil"
- Contains "don omar" -> "don_omar_bcn"
- Otherwise -> "unknown" (log a warning)

## Error Handling

If a Meta API call fails:
1. Read the error: `{"error":{"message":"...","type":"...","code":NNN}}`
2. Auth errors (code 190): re-read token from `../.env.local` before retrying.
3. Rate limits (code 32, 4, or 17): wait 60 seconds, retry once.
4. Not found (code 100): verify the ID by listing campaigns first.
5. Permission error (code 10, 200): report to Jaime, don't retry.
6. After fixing, retry the failed step -- don't start over.
7. **Never expose raw API error JSON** -- diagnose and summarize.
