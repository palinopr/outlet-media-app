## Budget Tracking

### Pull total spend for a client (by campaign name prefix)
```bash
# Get all campaigns with name containing "Zamora"
curl -s "https://graph.facebook.com/v21.0/act_787610255314938/insights?fields=spend&date_preset=this_month&filtering=%5B%7B%22field%22%3A%22campaign.name%22%2C%22operator%22%3A%22CONTAIN%22%2C%22value%22%3A%22Zamora%22%7D%5D&access_token=$TOKEN"
```

### From Supabase (for historical)
```bash
curl -s "${SUPABASE_URL}/rest/v1/meta_campaigns?select=campaign_id,name,spend,roas,daily_budget,start_time&name=like.*Zamora*&status=eq.ACTIVE" \
  -H "apikey: ${SUPABASE_KEY}" \
  -H "Authorization: Bearer ${SUPABASE_KEY}"
```

**Supabase column naming (important):**
- `meta_campaigns`: name column is `name` (NOT `campaign_name`). Budget is `daily_budget` (in cents).
- `campaign_snapshots`: spend column is `spend` (value is in cents despite the name). `roas` is a float.
- `agent_tasks`: orchestration ledger uses `from_agent`, `to_agent`, `action`, `status`, and `created_at`.

### Snapshot query (for trend analysis)
```bash
curl -s "${SUPABASE_URL}/rest/v1/campaign_snapshots?campaign_id=eq.{ID}&order=snapshot_date.desc&limit=7" \
  -H "apikey: ${SUPABASE_KEY}" \
  -H "Authorization: Bearer ${SUPABASE_KEY}"
```

### EATA/Vivaticket query (Don Omar BCN)
```bash
curl -s "${SUPABASE_URL}/rest/v1/tm_events?select=name,tickets_sold,tickets_sold_today,revenue_today,avg_ticket_price,gross,date,status&tm_id=eq.eata_14948" \
  -H "apikey: ${SUPABASE_KEY}" \
  -H "Authorization: Bearer ${SUPABASE_KEY}"
```

### Budget cap tracking

Some clients have total spend caps (e.g., Alofoke had $2K cap). When tracking:
- Calculate: total_spend / budget_cap = % consumed
- Alert at 80%, 95%, and 100% thresholds
- Project remaining days: (budget_cap - total_spend) / daily_budget
- Post alert to dashboard if approaching cap

**Currency note:** Don Omar BCN revenue is in EUROS (Vivaticket/EATA). All other clients are in USD (Meta). Never mix currencies in reports.

## Show Proximity Awareness

When preparing client updates, always check upcoming show dates. Shows within 3 days need special attention:
- Verify campaign is ACTIVE and spending normally
- Check ROAS — below 2.0 within 3 days of show is a flag
- After show date passes, verify campaign is PAUSED with $0 new spend
- **Post-show auto-pause rule:** If a campaign is still ACTIVE after its show date, flag it immediately — no confirmation needed.

Query upcoming shows:
```bash
curl -s "${SUPABASE_URL}/rest/v1/tm_events?select=name,date,status,venue_name,client_slug&date=gte.$(date +%Y-%m-%d)&order=date.asc&limit=10" \
  -H "apikey: ${SUPABASE_KEY}" \
  -H "Authorization: Bearer ${SUPABASE_KEY}"
```

## Alerts Endpoint

Post alerts to the dashboard when client-facing issues are detected:

```bash
INGEST_URL=$(grep INGEST_URL .env | cut -d= -f2)
INGEST_SECRET=$(grep INGEST_SECRET .env | cut -d= -f2)
BASE_URL="${INGEST_URL%/api/ingest}"
curl -s -X POST "$BASE_URL/api/alerts" \
  -H "Content-Type: application/json" \
  -d "{\"secret\":\"$INGEST_SECRET\",\"message\":\"[alert text]\",\"level\":\"warning\"}"
```

Level must be one of: "info", "warning", "critical" (Zod-validated — "warn"/"error" will be rejected).
